## 1. System summary

KISS is a Python CLI installer that scaffolds a target user project for
Spec-Driven Development (SDD). All intelligence is in (a) the assets
shipped in the wheel and (b) the *integration* layer that knows how to
deploy those assets to the directory layout each AI provider expects. At
runtime KISS itself is short-lived: it reads inputs, writes files, and
exits. The AI providers it installs for are external processes invoked by
the user *after* `kiss init`.

Design themes evident in the code:

- **Offline-only after install** — the wheel embeds every preset,
  extension, workflow, agent skill, and the integrations catalog
  (`pyproject.toml:33-42`, `_bundled_catalogs.py:1-12`,
  comment in `_locate_core_pack`,
  `installer.py:301-315`). `tests/test_offline.py` defends this.
- **Single entry, fan-out installers** — one `kiss init` invocation
  runs N integrations in sequence (`cli/init.py:293-326`).
- **Hash-tracked installs** — every file written carries a SHA-256 in a
  per-integration manifest, enabling safe diff-aware uninstall/upgrade
  (`integrations/manifest.py:50-80`, `cli/integration.py:319-343`).
- **Multi-AI parity** — 7 integrations registered
  (`integrations/__init__.py:67-81`), all sharing the same
  install contract (`integrations/base.py:56-80`). Narrowed
  from 13 to 7 on 2026-04-26 per ADR-018 (see §4 below).

## 2. Containers (process / runtime boundaries)

KISS has only one process at a time, but several distinct *runtime
contexts* with their own contracts.

### C1 · `kiss` CLI process (Python 3.11+)

- Entry point: `src/kiss_cli/__init__.py:407-409` (`main()`),
  registered as `kiss` in `pyproject.toml:23`.
- Built atop Typer + Click — `pyproject.toml:7-8`,
  `src/kiss_cli/cli/__init__.py:11-18`.
- Reads project state from `.kiss/context.yml` (`context.py:155-172`)
  and `.kiss/init-options.json` (`config.py:22-33`).
- Writes:
  - the project tree under whichever folder the user picks, plus
  - per-integration directories under `<integration.config["folder"]>/`
    (e.g. `.claude/`, `.cursor/`, `.gemini/` …) —
    `integrations/base.py:284-301`.

### C2 · Hatch build hook (build-time, not runtime)

- `scripts/hatch_build_hooks.py:104-144` — runs at `uv build` time. Its
  `initialize()` method:
  1. Wipes `build/core_pack/` (`hatch_build_hooks.py:113-115`).
  2. Mirrors repo-root asset trees into it
     (`hatch_build_hooks.py:118-127`, with the `ASSET_MAP` at lines
     32-48).
  3. Shells out to `scripts/generate-checksums.py` to produce
     `build/core_pack/sha256sums.txt` (`hatch_build_hooks.py:129-138`).
- The wheel `force-include` then maps `build/core_pack` →
  `kiss_cli/core_pack` — `pyproject.toml:41-42`.

### C3 · `core_pack` static asset bundle (inside the wheel)

- Located at `kiss_cli/core_pack/` once installed; resolved at runtime
  by `installer.py:301-315`.
- Contains: `agent-skills/`, `subagents/`, `presets/`, `extensions/`,
  `workflows/`, `integrations/catalog.json`, plus `sha256sums.txt`.
- Integrity is verifiable by `_integrity.verify_asset_integrity()` at
  `_integrity.py:24-79` (asset corruption raises
  `AssetCorruptionError`).
  *Note*: `_integrity.verify_asset_integrity` is defined but I did not
  find a call site that invokes it during normal `kiss init`
  (**unverifiable — confirm**; logged as ANALYSISDEBT-002).

### C4 · Installed agent skills (per-AI runtime targets)

After `kiss init`, every selected integration owns a folder in the user
project, e.g.:

- `.claude/skills/kiss-<name>/SKILL.md` — `integrations/claude/__init__.py:42-54`
- `.github/skills/kiss-<name>/SKILL.md` — `integrations/copilot/__init__.py:34-39`
- `.gemini/commands/kiss.<name>.toml` — `integrations/gemini/__init__.py:8-21`
- `.cursor/skills/kiss-<name>/SKILL.md` — `integrations/cursor_agent/__init__.py:13-26`
- `.agents/skills/kiss-<name>/SKILL.md` — `integrations/codex/__init__.py:14-27`
- (and the remaining supported integrations) — see `integrations/__init__.py:50-81`.

These are not processes; they are static text files consumed by each
external AI tool's own runtime.

### C5 · External AI provider CLI/IDE processes

- The 7 supported integrations declare CLI install URLs

### C6 · GitHub Actions CI runner

## 5. Data flow — `kiss init my-project --integration claude`

1. Typer parses argv; `init()` runs at `cli/init.py:77-127`.
2. Project path resolved (`cli/init.py:142-184`); user is prompted only
   if no integration is provided (`cli/init.py:185-198`).
3. Tools precheck: `git`, then the primary AI's CLI binary —
   `cli/init.py:230-253`. Special-cased for Claude's local install paths
   in `installer.py:84-89`.
4. `StepTracker` rendered live — `cli/init.py:262-282`.
5. For each selected integration:
   - A new `IntegrationManifest` is created
     (`cli/init.py:294-298`).
   - `integration.setup(...)` runs (`cli/init.py:299-304`); the base
     class drives template processing
     (`integrations/base.py:771-817` for `IntegrationBase.setup`,
     `:892-955` for `MarkdownIntegration.setup`).
   - For each skill template, `process_template()`
     (`integrations/base.py:682-769`) extracts the script command from
     YAML frontmatter, replaces `{SCRIPT}`, `{ARGS}`, `__AGENT__`,
     `__CONTEXT_FILE__`; `bundle_skill_assets()` copies the skill's
     `scripts/` and `templates/` next to it —
     `skill_assets.py:79-127`.
   - `manifest.save()` writes `.kiss/integrations/<key>.manifest.json`.
6. `_install_shared_infra()` makes `.kiss/`
   (`installer.py:393-410`).
7. `install_custom_agents()` deploys role-agent prompts from
   `subagents/` into each integration's agent dir —
   `installer.py:429-494`.
8. `create_context_file()` writes `.kiss/context.yml` —
   `context.py:8-152`.
9. Optional `git init` + commit — `installer.py:126-149` invoked from
   `cli/init.py:351-398`.
10. Bundled `git` extension auto-installed (`cli/init.py:372-393`).
11. Bundled `kiss` workflow auto-installed (`cli/init.py:402-431`).
12. `ensure_executable_scripts()` chmods `*.sh` files
    (`installer.py:497-555`).
13. `save_init_options()` persists the chosen integrations + version
    (`cli/init.py:441-455`, `config.py:10-19`).
14. Optional preset install (`cli/init.py:466-489`).
15. Final summary tree + Next-Steps panel (`cli/init.py:512-599`).

Inputs and outputs at each step are file-based, not network. The whole
flow is offline once `uv tool install` is done — guard tested by
`tests/test_offline.py`.
