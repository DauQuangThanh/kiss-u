## 3. Circular-import notes

The codebase has known import-order pitfalls that the current
structure works around rather than eliminates:

- `kiss_cli/__init__.py:404` imports `from kiss_cli.cli import app`
  *after* `AGENT_CONFIG` has been built. The CLI submodules import
  `from kiss_cli import AGENT_CONFIG`, so this ordering is load-bearing
  (`cli/__init__.py:38-46` documents the trade-off).
- `kiss_cli/agents.py:19-29` lazily builds `CommandRegistrar.AGENT_CONFIGS`
  via a `_build_agent_configs()` helper that imports
  `kiss_cli.integrations` *inside* the function. The class' `__init__`
  catches `ImportError` (`agents.py:55-59`) so a circular load doesn't
  crash module init.
- `kiss_cli/installer.py:39-45` lazy-loads `StepTracker` for the same
  reason.
- `kiss_cli/workflows/engine.py:43-50` defers step-type imports inside
  `_register_builtin_steps()` to break the obvious circle between the
  engine and step modules.

These are workarounds, not real cycles in the strict sense ŌĆö Python
imports succeed because the deferred imports happen *after* the module
that would have caused the cycle has finished loading. No genuine
circular-import errors were observed during the scan. **(unverifiable ŌĆö
confirm by running the test suite; ANALYSISDEBT-007.)**
