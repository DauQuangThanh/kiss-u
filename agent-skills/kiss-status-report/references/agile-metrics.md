# Agile metrics — quick reference

For agile/scrum and kanban teams, include these metrics in the
status report's "Summary" or as a dedicated section.

| Metric | What it measures | How to use in a status report |
|---|---|---|
| **Velocity** | Features / story points completed per sprint | Forecast remaining work: remaining points ÷ velocity = sprints-left |
| **Burndown** | Work remaining over time within a sprint | Flag sprints where burndown stalls mid-period |
| **Burnup** | Work completed over time (plus total scope) | Shows scope creep visually; upward curve with stable total is healthy |
| **Cycle time** | Time from "in progress" → "done" for a single task | Report trend: improving / flat / worsening |
| **Lead time** | Time from "requested" → "delivered" | Compare to agreed SLA if one exists |
| **Sprint health** | % tasks completed, % velocity predictable, # blockers | One-line dashboard for sprint-level reports |
| **WIP** | Count of in-progress items | Over the WIP limit → flag as a process risk |
| **Throughput** | Items finished per week (Kanban-friendly) | Stable throughput + short cycle time = healthy flow |

## Reporting cadence defaults

- **Scrum team** — report per sprint (every 2 weeks), summarising the
  sprint goal, outcome, velocity, and next sprint's goal.
- **Kanban team** — report weekly or bi-weekly, focused on flow
  metrics.
- **Mixed / hybrid** — report per milestone for the waterfall parts,
  per sprint for the agile parts, combined in one period report.

## When to escalate via status report

- Velocity drops >25% for two consecutive sprints (team health
  concern).
- Cycle time doubles without explanation (process concern).
- WIP count persistently exceeds limit (flow concern).
