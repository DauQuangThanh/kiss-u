# Status report — what to include, what to cut

## Include

- **RAG + one-sentence reason** — sponsors skim headers first.
- **Accomplishments that change the narrative** — delivered
  features, closed risks, signed-off milestones.
- **Planned activities for the next period** — concrete, not
  aspirational.
- **Blockers with owner + ETA** — not a problem list; it's a
  request-for-action list.
- **Any change-request or decision the sponsor needs to see** —
  preview what's coming, do not surprise them.
- **Budget variance** — even if small; establishes a trend line.

## Cut

- **Activity lists** ("attended meeting X, drafted Y") — belongs
  in a standup log, not a status report.
- **Long lists of minor completions** — group into one bullet.
- **Jargon without context** — spell out acronyms on first use.
- **Risks that are not new or not escalating** — they live in the
  register; the status report only mentions Red + newly-Amber.
- **Opinion without data** — "team morale is low" must be backed
  by a signal (velocity drop, attrition, survey result).

## Length

- One page if possible. Two pages max.
- If more is needed, move detail into appendices or linked
  artefacts; the core report stays short.
