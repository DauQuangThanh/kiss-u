# Technology evaluation — dimensions to compare

Not every dimension matters for every topic; pick the 4–6 that
actually drive the decision.

| Dimension | Typical signals |
|---|---|
| **Maturity** | Release history, active contributors, known production users |
| **Fit for purpose** | Natural for the workload (e.g. relational → SQL) |
| **Operational burden** | Managed service vs. self-host; on-call volume |
| **Cost signal** | Licence, per-unit usage, infra, staff required |
| **Ecosystem** | Libraries, tooling, community knowledge |
| **Team skill** | Can the team operate it today? Hiring market |
| **Licence / legal** | GPL vs. permissive; export controls; patent grants |
| **Compliance** | Data-residency, certifications (SOC 2, ISO 27001, HIPAA) |
| **Performance** | Latency, throughput, scalability at stated load |
| **Portability** | Lock-in level; viable exit strategy |
| **Observability** | Native telemetry; integration with existing stack |
| **Security** | CVE history, hardening guides, default auth posture |

## AI-authoring note

- Always record the **source + date** for every factual claim.
- Flag any claim you can't verify as `(unverified — confirm)`.
- Prefer vendor docs + reputable third-party comparisons over
  marketing pages.
